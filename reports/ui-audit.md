# UI-аудит (перед редизайном)

Дата: 2026-06-21. Аудит проведён по `src/app`, `src/components`, `src/app/globals.css`, `src/lib/*`, `public/art`.

## 1. Где сейчас показываются завершённые записи

Это главная проблема, обнаруженная аудитом — фильтрация по статусу `completed` разбросана по проекту и неполна:

- **`src/app/archive/page.tsx`** — отдельная публичная страница, целиком посвящённая завершённым записям (`computeStatus === "completed"`). Подлежит удалению/редиректу.
- **`src/components/SiteHeader.tsx`** — пункт навигации «Архив» (`/archive`) присутствует и на мобильных, и на десктопе (одно и то же меню, мобильного варианта вообще нет).
- **`src/components/RecordExplorer.tsx`** (используется на `/games/[gameId]`, `/ending-soon`, `/archive`):
  - Фильтр «Статус» включает `completed` как полноценную опцию (`RECORD_STATUSES` содержит все три статуса) — пользователь может явно выбрать «Завершено» на странице игры или «Скоро закончатся».
  - Сортировка `recently-ended` показывает завершённые записи первыми — доступна как опция всегда, в т.ч. на страницах, где завершённые в принципе не должны быть видны.
  - Блок «Требует проверки» (`review`) строится из `filtered`, который берётся из `computed` **без отбора по статусу** — то есть завершённая запись с `confidence: conflicting/unverified` попадёт в «Требует проверки» наравне с активной/предстоящей.
- **`src/app/games/[gameId]/page.tsx`** — передаёт в `RecordExplorer` **все** записи игры (`gameRecords`), включая завершённые; фильтрация полностью на клиенте через `RecordExplorer`, то есть в момент SSR/до гидратации завершённые записи присутствуют в данных, переданных компоненту (не отфильтрованы на сервере).
- **`src/components/CalendarView.tsx`** — строит карты `starts`/`ends` из **всех** переданных записей без проверки статуса; при просмотре прошлых месяцев календарь показывает маркеры начала/окончания завершившихся записей. Календарь получает `records` без какой-либо предварительной фильтрации (`CalendarPage` передаёт `records` целиком).
- **`src/app/page.tsx` / `HomeDashboard.tsx`** — `activeCount`/`upcomingCount` в hero вычислены корректно по статусу, но `HomeDashboard` вычисляет статусы заново через `computeRecords(records, now)`, где `records` — это **все** записи (включая завершённые); секции дальше фильтруют по `status === "active"`/`endsToday`/`endsWithinDays`, что **технически верно отсекает завершённые**, но логика разбросана по компоненту вручную (`r.status === "active"`, `endsToday`, `endsWithinDays`) без единой функции видимости — высокий риск регрессии при следующих изменениях.
- **`src/app/records/[id]/page.tsx`** — открывает **любую** запись по id независимо от статуса, включая полностью завершённые. Прямая ссылка на завершённую запись отображает полноценную карточку с описанием, наградами и т.д.
- **Счётчики на `/games`** (`GamesPage`) считают `total` (все записи) и `active`, но не учитывают `upcoming` — пользователь не видит на плитке игры предстоящие записи, только активные и общее число (которое включает завершённые).

**Вывод:** фильтрация выполняется вручную и непоследовательно (`computeStatus(r, now) === "active"` в одном месте, `r.status === "completed"` в другом, нигде нет единой функции «можно ли показывать запись»). Это и есть основной технический долг, который нужно устранить централизованной функцией видимости.

## 2. Визуальные проблемы

- Карточка записи (`RecordCard`) использует акцентную полосу слева (`before:w-1`) только для типа записи; статус, достоверность и тип смешаны в одной строке тегов без чёткой иерархии — при длинных русских названиях события (частый случай в данных) строка тегов и заголовок визуально конкурируют.
- Hero-блок на главной (`src/app/page.tsx`) — фон через `background-image: url(/art/hero.svg)` и градиентный оверлей, но дальше нет чёткой структуры: только бейджи активно/впереди, нет числа игр и «скоро заканчивается» как отдельного значимого числа, нет кнопок-переходов (CTA) к календарю/актуальным событиям, как того требует обновлённый дизайн.
- Цветовая палитра завершённого статуса (`--color-completed: #94a3b8`) используется в `Badges.tsx`/`Countdown.tsx`, но после скрытия завершённых записей эти стили становятся мёртвым кодом, если не остаются как часть внутренней модели (тип `RecordStatus` всё равно должен содержать `completed`).
- Нет согласованной шкалы теней/радиусов как явных CSS-переменных — `card` использует `border-radius: 1rem` напрямую, без переменной; разные компоненты используют `rounded-lg`/`rounded-xl`/`rounded-2xl`/`rounded-3xl` без единого правила, когда что использовать.
- Контраст вторичного текста (`--color-muted: #9aa5c8` на `--color-bg: #080b16`) в целом приемлем (~7.5:1), но мелкий текст `text-xs` в сочетании с `text-muted` на полупрозрачных поверхностях (`bg-surface-2/60` и т.п.) местами визуально слабый — нужно перепроверить на карточках с `line-clamp`.
- `GameCover`/`GameAvatar` используют фон через CSS `background-image`, поэтому при длинных названиях игр (`GODDESS OF VICTORY: NIKKE`) заголовок в `h2`/`h3` обрезается через `truncate`, что само по себе нормально, но нигде не показан полный текст по hover/focus (нет `title`).

## 3. Проблемы навигации

- Пункт «Архив» в шапке ведёт на страницу, которая по новым требованиям не должна существовать как публичный раздел — требуется удаление пункта и редирект маршрута.
- Нет мобильного меню: на узких экранах `nav` в `SiteHeader` просто переходит на новую строку (`flex-wrap`) — пять пунктов меню в любом случае помещаются, но при добавлении доп. действий (например, кнопки в hero) шапка станет визуально перегруженной на маленьких экранах. Отсутствует паттерн disclosure/гамбургер, который явно требуется в новом ТЗ.
- Подвал не содержит ссылки на GitHub, предупреждения об изменении дат и пояснения о независимости от разработчиков игр — упомянуты в новых требованиях, отсутствуют сейчас.

## 4. Проблемы мобильной версии

- Сетка карточек (`grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`) корректно сжимается до одной колонки, но сама карточка не тестировалась на ширинах 320–375px с длинными русскими названиями — `line-clamp-2` на описании и `truncate` на заголовке игры спасают от переполнения, но строка тегов (`TypeBadge` + `SubtypeBadge`/`EventSubtypeBadge` + `ConfidenceBadge`) может перенестись на 2-3 строки и визуально «разбухнуть» карточку без видимых проблем вёрстки, но с потерей компактности.
- Календарь: сетка 7 колонок с `aspect-square` ячейками технически не переполняется по горизонтали, но на 320px ширина ячейки ~40px — текст числа + два мелких бейджа (↑N ↓N) может становиться нечитаемым; нет отдельного компактного представления для мобильных (например, списка дней вместо сетки или скролла).
- Фильтры в `RecordExplorer` — сетка `grid-cols-2 sm:grid-cols-3 lg:grid-cols-6` на маленьких экранах даёт 2 колонки на 6 полей (3 строки), что приемлемо, но области нажатия `<select>` не проверены на соответствие 40-44px.

## 5. Дублирование стилей

- Повторяющиеся классы Tailwind для бейджей-«пилюль» (`rounded-md px-2 py-0.5 text-xs font-semibold ring-1 ring-inset`) встречаются инлайн в `RecordCard.tsx` и `records/[id]/page.tsx` отдельно от компонентов `Badges.tsx` — два места задают одинаковый визуальный стиль для премиум-валюты/призывов без общего компонента.
- `formatInZone`/`offsetForTimezone`/регулярки `DATE_ONLY`/`LOCAL_DATE_TIME` **продублированы** между `src/lib/status.ts` и `src/lib/format.ts` (две независимые копии одной и той же логики смещения часового пояса) — риск расхождения при изменении списка поддерживаемых зон.
- Стили «карточки» (`card` utility) переопределяются точечно через доп. классы (`border-amber-400/30 bg-amber-400/5` для демо-уведомления) — паттерн рабочий, но нет общего набора «вариантов карточки» (info/warning) как переменных.

## 6. Проблемы карточек

- `RecordCard` не показывает обратный отсчёт и статус как два явно разных визуальных блока — `StatusBadge` в правом верхнем углу обложки, `Countdown` ближе к низу карточки; связь между ними неочевидна при быстром сканировании.
- Оригинальное название (`originalTitle`) не показано на карточке вообще (только на странице записи) — по новым требованиям должно быть видно вторичным текстом и на карточке.
- Источник (`Источник ↗`) показан как отдельный блок с бордером внизу карточки — но при отсутствии источника карточка теряет нижнюю границу, из-за чего карточки разной высоты в одном ряду визуально «прыгают».

## 7. План централизации фильтрации завершённых записей

1. Создать `src/lib/visibility.ts` с функциями `isRecordActive`, `isRecordUpcoming`, `isRecordPubliclyVisible`, `getVisibleRecords`, основанными на уже существующих `computeStatus`/`computeRecord`/`parseRecordDate` из `src/lib/status.ts` (без дублирования парсинга дат).
2. Все страницы (`page.tsx`, `games/page.tsx`, `games/[gameId]/page.tsx`, `calendar/page.tsx`, `ending-soon/page.tsx`, `records/[id]/page.tsx`) переводятся на `getVisibleRecords(records, now)` **на сервере**, до передачи данных клиентским компонентам — компоненты получают только видимые записи, а не весь набор с последующей фильтрацией на клиенте.
3. `RecordExplorer`/`HomeDashboard`/`CalendarView` перестают принимать «сырые» `TrackerRecord[]`, содержащие завершённые записи; внутренний `computeRecords` остаётся для тика времени на клиенте, но входной набор уже не содержит завершённых.
4. Из фильтра «Статус» и сортировки убираются опции, относящиеся к `completed` (само значение статуса остаётся в типах для внутренней модели, как и просили).
5. Блок «Требует проверки» строится только из видимых записей.
6. `records/[id]/page.tsx` вызывает `notFound()`, если вычисленная запись не проходит `isRecordPubliclyVisible`.
7. `/archive` заменяется на редирект на `/`.

Этот план реализуется в последующих задачах редизайна.
