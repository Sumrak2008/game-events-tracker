# Game Events Tracker — правила работы

Трекер игровых **баннеров, событий и сезонов**. Данные хранятся в JSON-файлах в `data/` и являются единственным источником правды. Базы данных и админ-панели нет — данные меняются правкой JSON (в т.ч. через Claude Code) с последующим пушем в GitHub и деплоем на Vercel.

## Архитектура (кратко)

- `data/*.json` — игры, баннеры, события, сезоны, реестр источников.
- `src/lib/` — типы (`types.ts`), загрузка данных (`data.ts`), вычисление статуса (`status.ts`), форматирование (`format.ts`), подписи (`labels.ts`).
- `src/components/` — UI. `src/app/` — страницы (App Router).
- `scripts/validate-data.mjs` — валидатор данных (`npm run validate`).
- Подробности формата записей — в [docs/json-format.md](docs/json-format.md).
- Реестр источников по играм — в [docs/data-sources.md](docs/data-sources.md).

## Модель данных (обязательно соблюдать)

- Типы записей строго: `banner`, `event`, `season`. Ничего больше (без промокодов, патчей, техработ, новостей, рейдов, Twitch Drops и т.п.).
- Обязательные поля: `id`, `type`, `gameId`, `title`, `description`, `startAt`, `endAt`, `timezone`, `region`, `sourceType`, `confidence`, `sourceUrls`, `verifiedAt`, `isDemo`.
- Опционально: `verificationNote`, `imageUrl`, `note`, `sourceUrl` (устар., вместо него `sourceUrls`).
- Для баннеров опционально `bannerSubtype` (`character`/`weapon`/`equipment`/`other`).
- Для событий опционально: `eventSubtype` (`major`/`challenge`/`login`/`web`/`limited-mode`/`reward-event`/`other`), `rewards` (массив строк), `rewardSummary`, `premiumCurrencyName`, `premiumCurrencyAmount`, `summonCurrencyAmount`, `requirements`, `claimEndAt`.
- Даты — ISO 8601 **с указанием часового пояса** (например `2026-06-19T22:00:00+08:00`).
- Статус (`upcoming`/`active`/`completed`) **не хранится** — он вычисляется по текущему времени, `startAt` и `endAt`. Никогда не добавляй поле статуса в JSON.

## Реестр источников (`data/sources.json`)

**Нет одного универсального источника для всех игр.** Для каждой игры есть отдельный реестр в `data/sources.json` (тип `GameSourceRegistry` в `src/lib/types.ts`) с тремя ролями источников:

1. **`discoverySources`** — источник обнаружения: где искать максимально полный перечень баннеров/событий/сезонов.
2. **`verificationSources`** — источник подтверждения: где проверять точные даты, регион и награды (как правило, официальные).
3. **`fallbackSources`** — резервный источник: используется, если основной недоступен или неполон.

Официальный сайт — не обязательно единственный источник; он имеет высший приоритет среди источников подтверждения, но специализированные сайты (Game8, Prydwen, GameWith, Wowhead, Nikke.gg и т.п.) разрешены и необходимы для полноты на этапе обнаружения. Полный список источников по каждой игре — в [docs/data-sources.md](docs/data-sources.md).

Перед использованием источника проверяй поле `enabled`: `enabled: false` означает, что источник временно не используется (например, точный URL не подтверждён) — не бери из него данные, пока не проверишь и не включишь.

Поисковые сниппеты источником НЕ являются — всегда открывай исходную страницу. Запрещены как подтверждение: фан-вики без указания источников, Reddit, форумы, агрегаторы, непроверенные списки, страницы с утечками и прогнозами будущих версий.

## Достоверность (`sourceType` + `confidence`)

- `sourceType`: `official | specialist | wiki | community` — категория самого сильного источника.
- `confidence`:
  - `confirmed` — дата подтверждена **официальным** источником (требует `sourceType: official`).
  - `corroborated` — дата совпадает минимум в **двух независимых** специализированных источниках.
  - `single-source` — найден только один доверенный специализированный источник (пониженная уверенность).
  - `conflicting` — источники дают разные даты/условия. В UI — блок «Требует проверки».
  - `unverified` — сведения найдены (например, через discovery-источник), но пока не подтверждены ни одним приемлемым источником. В UI — тот же блок «Требует проверки».
- `sourceUrls` — массив **всех** использованных ссылок (первая — сильнейшая).
- `verificationNote` — причина неопределённости / как сверялось.

### Правила полноты для каждой игры

1. Открыть источник(и) обнаружения для игры из реестра.
2. Составить полный список текущих и объявленных событий, баннеров и сезонов.
3. Открыть отдельную страницу каждой записи (не ограничиваться сниппетом).
4. Извлечь даты, регион, награды и требования.
5. Попытаться подтвердить запись источником подтверждения (официальным, если есть).
6. Если официального подтверждения нет — найти второй специализированный источник для `corroborated`, иначе `single-source`.
7. Сохранить прямые URL всех использованных страниц в `sourceUrls`.
8. Не прекращать поиск после проверки одного официального сайта — пройти всю цепочку: обнаружение → подтверждение → резерв.

## Правила работы с данными (СТРОГО)

1. **Используй реестр источников игры** (`data/sources.json` → `docs/data-sources.md`): сначала `discoverySources` для полноты перечня, затем `verificationSources` (официальные источники имеют высший приоритет) для точных дат/наград, и только при необходимости `fallbackSources`.
2. **Не придумывай даты** и не выдавай предположение за факт. Точное количество наград не выдумывай: если подтверждён только факт премиальной валюты — укажи `premiumCurrencyName` без количества.
3. **Сторонние данные перепроверяй:** ищи минимум два независимых специализированных источника; сравнивай название, даты начала/окончания, регион/сервер; сохраняй все ссылки в `sourceUrls`. Если найден только один — `single-source`; если ни один — `unverified`.
4. **При конфликте дат** не выбирай одну произвольно — ставь `confidence: conflicting`, объясняй в `verificationNote`. Записи `conflicting` и `unverified` в UI идут в блок «Требует проверки», не среди подтверждённых.
5. **Учитывай регион и часовой пояс**: заполняй `region` и `timezone`, даты — с корректным смещением.
6. **Не удаляй завершённые записи.** Архив формируется автоматически по прошедшему `endAt`.
7. **Не заменяй подтверждённое неподтверждённым.** `confirmed`/`corroborated` нельзя ухудшать до `single-source`/догадки.
8. **`isDemo: true`** для демонстрационных и ориентировочных записей (в UI — пометка «Демо-данные»). Не выдавай вымышленное за реальное.
9. **Изображения с сайтов игр не копируй.** Используется оригинальный SVG-арт (`public/art/games/<id>.svg`); `imageUrl` — задел под собственные растровые изображения.
10. **Полнота:** не останавливайся, если на офиц. сайте нет данных — пройди всю цепочку источников. Собирай не только баннеры, но и временные события с наградами (премиум-валюта, билеты призыва, персонажи и т.п.) и сезоны. Постоянный контент без даты окончания не добавляй.
11. **Перед публикацией показывай отчёт об изменениях** и жди подтверждения.
12. **Не публикуй при ошибках проверок** (`validate`/`lint`/`typecheck`/`build`).

## Проверки перед публикацией

```bash
npm run format      # Prettier
npm run lint        # ESLint
npm run typecheck   # tsc --noEmit
npm run validate    # валидация JSON
npm run build       # production-сборка
```

Все пять должны проходить. Подробности публикации — в [docs/publishing.md](docs/publishing.md).
